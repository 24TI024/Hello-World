##### ソフトウェア工学2026  

# 第1回  
　ソフトウェア工学を学んでいく差異に、難しいところとしては、学んでいた事に対して、実践したときの感触はかなり違うので、実際にやってみないと分からないことが多い。また、複数人で開発を進めて行くことが多いので、知識の量を全員に共有する必要がある。  
# 第2回  
　では、ソフトウェアとは何かと言うと、「情報」を扱うプロダクトそのもの又は、プロダクトを提供する手段である。  
　また、ソフトウェアは以下のように定義される。  
- 実行される事により、必要な特性や機能、性能を提供する命令群。 
- プログラムが適切に情報を扱うことを可能とするデータ構造  
- プログラムの操作や、使用法を記述した情報   

これらのソフトウェアの特徴として、進化し続ける環境でも適切なサービスを提供できるように、再構築されなければならない。  
　では、ソフトウェア工学のゴールとは、ソフトウェアシステムが常に変化し続ける状況に対応出来る方法を見つけることであることと、全てのソフトウェアは古いシステムから構築され、さらに互換性を持ち、相互運用できることを目指している。  
　そんなソフトウェア工学の定義は、ソフトウェアを一定の規則に基づき、管理できるようにすることである。  
　ソフトウェア工学が世間にどのように必要とされているのだろうか。例えば、現代社会では、あらゆるものがデジタル化されているので、ソフトウェアを効率良く開発し、よりよいものにしていく必要がある。他にも、そもそもIT人材が不足しているので、育成するという意味でも、ソフトウェア工学は必要である。さらに、ライフラインなどの多様なソフトウェアが登場したことにより、不測の事態に対応出来る力や、ニーズにあったソフトウェアがとても大事になってくる。また、ユーザーは、ソフトウェアが持つ機能を自由に使うことが出来るので、観測可能な範囲で危険な挙動をしないか気をつけておかなければならない。  
　ソフトウェア開発をする際に気をつけなければいけないことが、リソースは有限であるので、スケールする必要がある。例として、人件費やコンピュータ、メモリやストレージ、開発者の時間などである。これらを区分けすると、
- 資金などを指す財務コスト
- CPU時間を指す資源コスト
- エンジニアリング労力を指す人件コスト
- 行動を取る際にかかるコストを差す処理コスト
- 行動を取らない際にかかるコストを差す機会コスト
- 選択の影響を指す社会コストがある。  

　問題解決をする際には、問題を理解し、解決策を計画し、実行した後、結果を確認し、正しいことを理解するという流れを用いると良い。これをソフトウェア開発になぞらえると、まず依頼人とのコミュニケーションを取り、問題分析することで、作るべきソフトウェアを考える。その後、自分の考えをもとに、モデリングとソフトウェア設計を行い、設計をもとにコードを実装する。その後、計画が正しいかを実行し、エラーが無いか、求めた出力をするかを調べる。  
　こうしたソフトウェア開発のなかで大切にすべきものとして、Hookerの7原則というものが存在する。このHookerの7原則では、ユーザーに提供する上での分かりやすさ、開発する上での心構え、将来の開発者が設計を理解しやすくなるようにすべきである、といった内容が語られている。  

# 第3回

　ソフトウェアライフサイクルとは、ソフトウェアの構想を開始してからソフトウェアの廃止までの流れのことである。  
　計画段階では、まず要件定義をする。要件定義で大事なことは、実現する、しない機能を明確にし、スケジュールを立てていく。要件定義をすることにより、依頼人とのやりとりはスムーズになるほか、協力を仰ぐことが容易になる。こうして作った要件定義を元に、コーディングが出来るレベルの仕様書に落とし込んでいく。この際に、チームの役割分担を示したWBSを作成しておく。  
　製作段階では、先ほど作成した設計書に従い、システム構築を進めていく。こうして完成したソフトウェアにたいしてデバッグを進めていくのだが、完全にバグを潰すことは難しいので、実際に使って発生するかどうかテストしておくことが重要である。また、外注する場合、メンテナンスを自分たちでするかどうかにより、ソースコードを納品してもらうかどうかが変わる。また、外注先を、賃金の安い海外企業に委託する、オフショア開発という手法があるが、文化や言語、品質に対する意識の違いに気をつけることが大事である。
　開発方法の一つとして、ドキュメンテーションというものがある。これは複数人で非同期のソフトウェア開発を進めていくものである。この開発方法の注意点として、海外企業とのプロジェクトが多く、英語でのコミュニケーションが多いことに注意する必要がある。  

# 第4回

　プロジェクトとは、必ず終わりがある、独自の目的を達成するような仕事である。  
　目標視点の考え方は、フォアキャスティングとバックキャスティングがある。  
　フォアキャスティングは、どれだけやったかの視点で考えるもので、今の立ち位置を確認してから順に目標に進んでいく。特徴として、目標に到達できない可能性があることと、目標と遠いところに到達する可能性がある点があるので、注意する必要がある。  
　バックキャスティングは、何を成し遂げたかの視点で考える物で、定めた目標に向けて、何が必要かを計画し達成に向かっていく。特徴として、目標を明確に設定する必要がある。ただし、目標は具体的かつ達成する価値のるものにする点と、目標は適宜修正可能である。  
　ソフトウェアを見える化するための手法として、以下のものがある。  
-ソフトウェアの物量に関する分析
-ソフトウェアの実効性能に関する分析
-ソフトウェアの開発工数に関する分析
-ソフトウェアの品質に関する分析

がある。  
　品質、コスト、納期、といったパラメータをQCDに使用する。  
　QCDでは目的に合わせた優先度を設定する。品質を優先するときは、人命に影響があるようなシステムであり、費用を優先するときは、公共に関連するシステムであり、納期を優先するときは、イベント行事に関連するシステムである。このように、3つのパラメータを天秤に掛けながら開発を進めていく。  
　ソフトウェアの評価基準として、コードのステップ数とオブジェクト容量、ファンクションポイント法、使い勝手がある。  
　ソフトウェアのリリースタイミングはとても難しいが、*Hyrum*の法則に従い、ユーザーが予期せぬ危険に巻き込まれないようにする必要がある。  
　開発の手法として、様々なものが挙げられる。  
　ウォータフォール型開発プロセスとは、開発過程を上流から下流へ順番に進めていく方法である。進捗管理が容易であり、成果物が明確である。一方で、後の工程にしわ寄せが集中するリスクがある。  
　スパイラルモデルとは、プログラム開発を小さく分割し、フェーズごとにプロトタイプによるデモンストレーション→フィートバックの流れを実行する。一方で、プロトタイプ作成に想定外の作業量が発生するリスクあり。  
　反復型開発プロセスとは、ソフトウェアを機能分割し、これを「反復」と呼ぶ単位で管理するプロセスである。メリットとして、部分的に完成させていくので、顧客の要求を取り入れやすく、部分的な納品が可能である。  
　一方で、分割のための作業や管理業務が増える上に、全体像が見えづらい。  
　アジャイルプロセスとは、変化に対応出来る、柔軟なプロセスである。  
　このように、プロセスは色々あり、一長一短なので、適切に選ぶことが大切である。  

# 第5回・第6回 

 　WBSとは、*Work Breakdown Structure*の略称である。これは、プロジェクトを細かく分解し、一つの図にする方法である。  

 　これをするメリットとしては、  
- スコープが明確になる
- 作業予定が一目瞭然になる。
- プロジェクトを実施するときには、WBSを見て実行するだけで良い。

といったものがある。   

　作成方法は、
- スコープを明確にする。
- 大きな作業に区分けした後、さらに細かく分割していく。
- 分割した作業同士の関連を考える。
- 各グループの作業を洗い出す。

といった流れで実行する。

# 第7回

　コードを作成する際には、自分含め、誰もが見やすいと感じるようにしなければならない。それは、コードは書くときよりも読まれるときの方が多いからである。そのため、見やすいコードを作成するためのルールが必要である。    
　ルールの一つとして、Pythonでは、PEP8というものがある。他にも、Google Style Guidesなどがある。  
　では、どんなルールがあるのかというと、コードの1行の長さを制限したり、スペースの配置を決定したり、文ごとに改行を強制したり、命名規則を決めたり、UTF-8をメインで使用するなどのルールがある。何よりも大事なのは、コメントが正しい情報を表示しているかどうかである。仮に間違った情報を表示している場合は、即刻修正するべきである。  
　このルールに則っているかを判断するツールとして、代表的なものはflake8やpvlintがある。  
　これらのツールを使うことにより、コードをより正確に作成することが出来る。

# 第8回

　バージョン管理とは、ファイルが変更された際、誰がいつどのような変更をしたかを把握するためのシステムである。2通りの方法があり、集中管理型と、分散管理型である。  
　集中管理型は、同時編集をすると、コンフリクトが起こりやすいのと、同期に時間がかかるという欠点がある。また、リモートリポジトリのみがバージョン管理をしているので、障害に弱い。  
　分散管理型は、各自がローカルリポジトリでバージョン管理をしているので、障害に頑健である。また、リモートリポジトリへのアクセス頻度が低いので、コンフリクトが発生しにくい。  
　分散管理型のバージョン管理システムとして、「git」が挙げられる。  
　gitとは、変更履歴が残り、変更した箇所に戻ることが出来、他人と共同編集が出来るソフトウェアである。  
　gitのシステムとして、「コミット」というものがある。コミットとは、いつだれがなにをどのように実行したのかメッセージを残すものである。コミットする際は、複数ファイルを同時に実行するか、一つのファイルで実行するかをユーザーが自由に決定できる。  
　gitが管理するプロジェクトのフォルダとして、レポジトリというものがある。レポジトリには2種類あり、個々のプロジェクト実行環境を指すローカルレポジトリと、共有の管理場所である、リモートリポジトリである。
　ローカルレポジトリでの流れとして、まずワークツリーで、変更し保存する。そのあと、コミットしたいファイルを登録し、ステージングエリアでgitで管理するファイルを登録し、コミットする。このときに変更された内容は、記録に残っており、原則変更削除が出来ない。  
　リモートレポジトリでの流れとして、ローカルレポジトリでコミットされた変更を、リモートレポジトリへ反映する。これをgit pushという。その後、リモートレポジトリにある内容を、ローカルレポジトリに反映する。これをgit pullである。

# 第12回

　girhubには、CIと呼ばれるプロセスと、CDとCDと呼ばれるプロセスがある。  
　CIとは「Continuous Intergration」の略であり、コード変更を共有リポジトリに頻繁に統合するプロセスである。自動手スおとリビルドを定期的に実行し、バグの早期発見と修正を可能にしている。CIの自動化により開発がスムーズに進行するようになる。　　  
　CDとは「Continuous Delivery」の略であり、コード変更をテスト環境や本番環境に自動的にデプロイするプロセスである。自動デプロイメントを組み込み、手動操作によるデプロイの必要性を排除できる。CDにより、ユーザーフィードバックを迅速に反映可能である。　
　CI/CDパイプラインは、ソース→ビルド→テスト→デプロイ→検証→モニタリングの流れで実行されている。また、GitHub Actionsを使用したCI/CDは、ワークフローの定義→トリガー→ジョブとステップ→アクション→ランナーの流れで実行されている。  
　CIでflake8を適用する際、インデントを使って階層構造を表現するが、スペースによるインデントを利用したいので、Tabは利用しない。また、キーと値のペアで情報を表現する。その際、「：」で区切り、リスト形式のデータは「-」で表現する。また、ymlファイウは、設定用ファイルである。各タグの機能として、「name」は、ワークフローの名前を定義し、「on」は、トリガーイベントを定義し、「jobs」は、"syntax-check"というジョブを定義する。
